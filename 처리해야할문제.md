### [ ] 2. 개인정보 처리방침과 실제 데이터 처리 일치

Microsoft Clarity가 앱 시작 시 초기화되지만 처리방침에는 Firebase Analytics만 기재되어 있다. 앱 이름, 수집 정보, 처리위탁 업체, 이용자 수집 중지 방법도 실제 앱과 일치하지 않는다.

- 영향
  - App Store 개인정보 표시와 실제 SDK 동작 불일치
  - 사용자에게 잘못된 개인정보 처리 정보 제공
- 대상
  - `apps/native/app/_layout.tsx`
  - `apps/native/utils/clarity.ts`
  - `apps/native/components/modal/privacy-modal.tsx`
  - `apps/native/components/modal/terms-policy-modal.tsx`
- 처리 기준
  - 서비스명을 `이루라`로 통일한다.
  - 이메일, 사용자 ID, 비밀번호, 닉네임, 직업, 소셜 로그인 정보, 기기 ID, 푸시 토큰, 사용자 이미지, 문의 등 실제 수집 항목을 반영한다.
  - Microsoft Clarity, 푸시 알림, 소셜 인증, 서버 호스팅 등 실제 처리위탁·제3자 SDK를 반영한다.
  - 개인정보 처리방침에 설명한 분석 데이터 수집 중지 기능을 앱에서 실제로 제공한다.
  - 동의 또는 설정 상태를 확인한 뒤 Clarity를 초기화한다.
  - App Store Connect의 개인정보 응답을 실제 SDK 동작과 동일하게 작성한다.
- 검증
  - 신규 설치에서 선택 전 Clarity가 시작되지 않는지 확인한다.
  - 수집 중지 후 세션 데이터가 전송되지 않는지 확인한다.

## P1 · 높은 위험

### [x] 3. 로그아웃 및 계정 전환 시 사용자 캐시 완전 삭제

완료 상태: 완료 (2026-07-22)

인증 사용자 ID가 바뀌면 React Query client를 재생성하고 기존 query·mutation cache를 제거하도록 변경했다. 통계, 알림 설정, 친구 요청, 퀘스트, 보상 캐시는 안정적인 `userId`로 분리했으며 A 계정 캐시가 B 계정으로 전달되지 않는 전환 테스트를 추가했다.

React Query의 QueryClient가 앱 수명 동안 유지되지만 로그아웃에서 캐시를 제거하지 않는다. 사용자 ID가 없는 쿼리 키가 있어 동일 기기의 다음 계정에 이전 계정 데이터가 노출될 수 있다.

- 대상
  - `apps/native/store/auth.store.ts`
  - `apps/native/app/_layout.tsx`
  - `packages/shared/components/provider/QueryProvider.tsx`
  - `packages/shared/hooks/useStat.ts`
  - 사용자별 통계, 알림 설정, 친구 요청, 퀘스트, 보상 관련 쿼리 키
- 처리 기준
  - 로그아웃 시 private query와 mutation cache를 제거한다.
  - 사용자별 데이터의 쿼리 키에 안정적인 `userId`를 포함한다.
  - 계정이 바뀌면 이전 QueryClient를 재사용하지 않도록 인증 범위에 맞춰 재생성하는 방식을 검토한다.
- 검증
  - A 계정에서 데이터를 조회한 뒤 로그아웃한다.
  - 네트워크를 차단하고 B 계정으로 로그인했을 때 A 계정 데이터가 한 프레임도 표시되지 않아야 한다.

### [x] 4. 어떤 오류에서도 로컬 로그아웃 완료 보장

완료 상태: 완료 (2026-07-22)

로그아웃 시 사용자 상태를 먼저 종료하고 토큰, 위젯, 공유 확장 정리를 `Promise.allSettled`로 격리했다. 푸시 토큰 API는 timeout이 적용된 공통 HTTP client로 통합했으며, 인증 갱신 실패와 갱신 후 재요청의 `401`에서도 토큰 삭제 및 로컬 세션 종료가 보장되도록 수정했다.

푸시 토큰 삭제에는 timeout이 없고, 위젯 또는 공유 확장 데이터 정리에서 예외가 발생하면 `user: null` 설정까지 도달하지 않는다. 인증 실패 콜백이 예외를 던지면 이후 인증 실패 처리도 막힐 수 있다.

- 대상
  - `apps/native/api/push-token.api.ts`
  - `apps/native/store/auth.store.ts`
  - `packages/shared/api/index.ts`
- 처리 기준
  - 푸시 토큰 API를 공통 HTTP client로 전환하고 timeout을 적용한다.
  - 토큰·사용자 상태·쿼리 캐시 제거는 `finally`에서 반드시 완료한다.
  - 위젯, 공유 확장, 서버 푸시 토큰 정리는 `Promise.allSettled` 등으로 격리한다.
  - `isRedirecting`은 콜백 성공 여부와 관계없이 `finally`에서 복구한다.
  - 갱신된 토큰으로 재시도한 요청이 다시 401이면 로컬 세션을 종료한다.
- 검증
  - 네트워크 무응답, SecureStore 실패, 위젯 모듈 실패, 공유 모듈 실패를 각각 주입해도 로그아웃 화면으로 이동하는지 확인한다.

### [x] 5. 소셜 access token의 라우트 전달 제거

완료 상태: 완료 (2026-07-22)

카카오와 Apple 신규 가입 credential을 provider, 인증 시도 ID, 원본 credential 만료 시각, 최대 5분 TTL과 결합해 일회성 메모리 저장소로 전달하도록 변경했다. `/social-sign-up` 라우트에는 파라미터를 전달하지 않으며, 유효한 로컬 인증 시도가 없는 직접 진입과 만료 시 로그인 화면으로 복귀한다. 성공, 인증 취소, 화면 이탈, 만료 시 임시 인증 정보가 삭제되도록 처리했다.

카카오 access token을 Expo Router 파라미터로 `/social-sign-up` 화면에 전달하고 있다.

- 대상
  - `apps/native/hooks/useAuth.ts`
  - `apps/native/app/social-sign-up.tsx`
  - `apps/native/store/pending-apple-auth.store.ts`
- 처리 기준
  - 카카오 인증 정보도 Apple과 같은 일회성 메모리 저장소로 전달한다.
  - 토큰을 URL, 라우트 파라미터, 로그, 분석 이벤트에 포함하지 않는다.
  - provider, nonce/state, 만료 시각을 인증 시도와 결합한다.
  - 유효한 로컬 인증 상태 없이 딥링크로 가입 화면을 직접 열면 로그인 화면으로 돌려보낸다.
  - 성공·취소·화면 이탈·만료 시 인증 정보를 즉시 삭제한다.
- 검증
  - 딥링크, 라우터 상태, 디버그 로그에 access token이 남지 않는지 확인한다.

### [x] 6. 공유 확장 이미지 입력 크기 제한

완료 상태: 완료 (2026-07-22)

공유 확장에서 이미지 3개, 파일당 10MB, 총 30MB, 원본 6천만 픽셀로 입력을 제한했다. Android는 64KB 버퍼로 스트리밍하고 초과·실패 시 부분 파일을 삭제하며, iOS는 파일 표현과 `CGImageSource` 메타데이터를 이용해 전체 파일 적재 없이 처리한다. JPEG·PNG·HEIC·HEIF·WebP만 허용하고 긴 변 4096px 이하 JPEG로 변환하며, 손상 파일과 provider 오류는 안전한 사용자 오류로 처리한다. JS의 기존 크기·픽셀 검사는 최종 방어선으로 유지했다.

Android는 외부 스트림을 `readBytes()`로 전부 읽고, iOS는 `Data(contentsOf:)`로 전체 파일을 메모리에 적재한다. JS의 10MB 제한은 네이티브 적재 이후이므로 OOM을 막지 못한다.

- 대상
  - `apps/native/plugins/with-routine-share-android.js`
  - `apps/native/targets/routine-share/ShareViewController.swift`
  - `apps/native/utils/request-image.ts`
- 처리 기준
  - 네이티브 입력 단계에서 MIME, 파일 크기, 이미지 픽셀 수를 검증한다.
  - Android는 제한된 버퍼로 스트리밍하며 상한을 넘으면 즉시 중단하고 부분 파일을 삭제한다.
  - iOS는 리소스 크기를 먼저 확인하고 `CGImageSource` 등을 이용해 다운샘플링한다.
  - 손상된 이미지와 ContentProvider 예외를 사용자 오류로 처리한다.
- 검증
  - 정상 이미지, 10MB 초과 이미지, 초대형 픽셀 이미지, 길이를 알 수 없는 스트림, 손상된 이미지로 공유 확장을 테스트한다.

### [x] 8. 인증 전 API의 `Bearer null` 전송 제거

완료 상태: 완료 (2026-07-22)

보호 API는 공백을 제거한 실제 access token이 있을 때만 `Authorization` 헤더를 전송하도록 변경했다. 이메일 확인·인증 요청·인증 확인·직업 목록을 공개 API로 분류했으며, 해당 API의 `401`에서는 토큰 갱신이나 로컬 세션 종료를 시도하지 않는다.

토큰이 없는 인증 전 요청에도 `Authorization: Bearer null`이 설정된다. 이메일 인증과 직업 목록 API가 public URL 목록에 포함되지 않았다.

- 대상
  - `apps/native/api/bootstrap.api.ts`
  - `packages/shared/api/index.ts`
  - `packages/shared/api/auth.api.ts`
- 처리 기준
  - 유효한 토큰이 있을 때만 Authorization 헤더를 설정한다.
  - 이메일 확인, 인증 요청·확인, 직업 목록의 인증 정책을 서버와 일치시킨다.
  - public API의 401에서는 refresh를 시도하지 않는다.
- 검증
  - 신규 설치 상태에서 회원가입 전체 흐름을 실제 API로 테스트한다.
  - 요청 캡처에서 `Bearer null`이 전송되지 않는지 확인한다.

### [ ] 9. 이메일 인증 폴링 직렬화 및 backoff 적용

2.5초 `setInterval`이 이전 요청 완료를 기다리지 않아 느린 네트워크에서 요청이 중첩된다. 일시 오류 후 정상 응답이 와도 UI가 오류 상태로 남을 수 있다.

- 대상
  - `apps/native/app/sign-up-email-verification.tsx`
- 처리 기준
  - 요청 완료 후 다음 `setTimeout`을 예약하는 직렬 폴링을 사용한다.
  - 지수 backoff, 최대 간격, jitter, 429 `Retry-After` 처리를 적용한다.
  - 정상 응답을 받으면 `waiting` 상태와 메시지를 복구한다.
  - 화면 이탈과 만료 시 진행 중 요청을 취소한다.
- 검증
  - 10초 지연 응답에서도 동시 요청이 한 개를 넘지 않아야 한다.

### [ ] 11. 전체 루틴 조회의 부분 성공 숨김 제거

`/routine/list/all`과 `/routine/list` 중 하나만 성공해도 완전한 목록으로 반환하여 중지·숨김 루틴 누락을 정상 상태로 오인할 수 있다.

  - fallback은 404 또는 명시적인 미지원 응답에서만 사용한다.
  - 부분 결과를 허용해야 한다면 반환 타입과 UI에서 partial 상태를 명시한다.
- 검증

- 공유 확장의 대형·손상 이미지 테스트를 추가한다.
- dependency audit 결과를 앱 런타임, 빌드 도구, 개발 도구로 구분하고 production 및 CI에 도달하는 Critical/High 취약점을 제거하거나 근거를 문서화한다.
- 실제 EAS production 빌드의 권한, URL scheme, 환경 변수, App Store 개인정보 응답을 최종 확인한다.
